# FreeLang v2 - AI 피드백 루프 철학

## 🧠 핵심 개념

**"AI는 학생이 아니라 설계자다"**

```
이전 패러다임:
AI는 오류 메시지를 읽고 배운다 (수동 학습)
❌ 비효율적, 느림, 반복됨

새로운 패러다임:
AI는 제안을 평가하고 피드백한다 (능동 학습)
✅ 효율적, 빠름, 자기주도적
```

---

## 🎯 3가지 핵심 인사이트

### Insight 1: "오류 = 기회"

```
이전:
입력 → 파싱 → ❌ 오류 발생
메시지: "헤더 정의 미흡"
AI: "아, 다음번엔 더 정확하게 작성해야겠네"
(수동 학습, 느림)

새로운:
입력 → 자동 분석 → 제안 헤더 제시
AI: "이 헤더 맞아? 아니면 수정할까?"
(능동 평가, 빠름)

→ AI가 오류를 "경험"하지 말고 "선택"하게 함
```

### Insight 2: "피드백 = 성장"

```
1차: "배열 더하기"
  → 제안: sum (90% 신뢰도)
  → AI: "네 맞아"
  → v2 학습: "배열 + 더하기 = sum (90%)"

2차: 다시 "배열 더하기"
  → 제안: sum (92% 신뢰도) ← 개선됨!
  → AI: "네"
  → v2 학습: "배열 + 더하기 = sum (92%)"

...

10차: "배열 더하기"
  → 제안: sum (98% 신뢰도) ← 거의 완벽
  → AI: "자동으로 sum으로 해도 돼"

→ 피드백의 누적 = v2의 진화
```

### Insight 3: "자율성 = 신뢰"

```
AI가 할 수 있는 것:
1️⃣ 제안 수락 (✅ 승인)
   "네, 이 헤더 맞아"
   → 즉시 코드 생성

2️⃣ 제안 수정 (✏️ 수정)
   "아니, 이렇게 변경해"
   → 수정된 헤더로 재분석

3️⃣ 다시 제안 (🔄 재제안)
   "다른 옵션 있어?"
   → 대안 헤더 3개 제시

4️⃣ 취소 (❌ 취소)
   "그냥 다시 입력할게"
   → 새로운 입력 대기

→ AI가 완전히 자율적으로 결정
```

## 🔄 피드백 루프의 4가지 시나리오

### Scenario 1: "좋은 제안" (95% 신뢰도)

```
입력: "배열 합산"
  ↓
제안: fn sum: array<number> → number
신뢰도: 95% ✅ (매우 높음)
  ↓
AI: [✅ 승인]
  ↓
결과: 코드 생성 진행
학습: "배열 + 합산 = sum (95%)" 저장
```

### Scenario 2: "중간 제안" (70% 신뢰도)

```
입력: "모든 수 더함"
  ↓
제안: fn sum: array<number> → number
신뢰도: 70% ⚠️ (중간)
  ↓
AI의 선택 1: [✅ 승인]
  → 맞다고 생각하면 코드 생성
  → 학습: "모든 수 + 더함 = sum (70%)"

AI의 선택 2: [✏️ 수정]
  → "아니, 이건 다른 거야"
  → 수정된 헤더로 재분석
  → 학습: "모든 수 + 더함 ≠ sum, 올바른 것은..."

AI의 선택 3: [🔄 재제안]
  → "다른 옵션 있어?"
  → 대안 제시
  → 가장 맞는 거 선택
```

### Scenario 3: "낮은 제안" (45% 신뢰도)

```
입력: "xxx"
  ↓
제안: fn foo: ??? → ???
신뢰도: 45% ❌ (낮음)
메시지: "이 입력은 의도가 불명확합니다"
  ↓
AI의 선택 1: [✏️ 수정]
  → 더 명확한 입력으로 수정
  → "배열 더하기"로 재입력

AI의 선택 2: [❌ 취소]
  → 처음부터 다시
  → 새로운 입력 대기
```

### Scenario 4: "AI의 자기지시" (Directive)

```
입력: "배열 더하기"
  ↓
제안 헤더:
fn sum: array<number> → number ~ "배열의 모든 수를 더하기"
reason: "통계 연산 기초"
directive: "메모리 효율성 우선"
  ↓
AI: "아니, 이건 빨라야 해"
[✏️ 수정] → directive를 "속도 우선"으로 변경
  ↓
결과: 다른 최적화 전략으로 코드 생성
학습: "sum + 속도 우선 = 캐싱 적용"
```

## 💾 피드백 저장 구조

```typescript
interface AIFeedbackRecord {
  session_id: string;
  timestamp: Date;

  // 입력 단계
  user_input: string;              // "배열 더하기"
  input_confidence: number;         // 입력 명확도 (0~100)

  // 제안 단계
  generated_header: string;         // fn sum: array<number> → number
  suggested_reason: string;
  suggested_directive: string;
  suggested_confidence: number;     // 95%

  // 피드백 단계
  feedback_type: "approved" | "edited" | "rejected" | "alternative";
  ai_response: string;              // AI가 제시한 수정사항

  // 결과 단계
  final_header: string;             // 최종 확정된 헤더
  final_reason: string;
  final_directive: string;

  // 학습 데이터
  confidence_improvement: number;   // 이전 vs 현재 신뢰도 차이
  learning_category: string;        // "pattern_match" | "type_inference" | ...

  // 메타데이터
  complexity: "O(n)" | "O(n²)" | ...;
  operation_type: "array" | "string" | "number" | ...;
}
```

## 🎓 학습 메커니즘

### Level 1: 직접 학습 (Feedback → Pattern Update)

```
피드백: AI가 "네, 맞아" 승인
  ↓
학습: 이 입력-의도 조합의 신뢰도 증가
  예: "배열 더하기" → sum (90% → 92%)
```

### Level 2: 간접 학습 (Related Patterns)

```
피드백: "배열 더하기" 승인
  ↓
학습 (간접):
- "수열 더하기" 신뢰도도 약간 증가
- "배열 합산" 신뢰도도 약간 증가
- "여러 수 더함" 신뢰도도 약간 증가

→ 유사 패턴들에도 긍정적 영향
```

### Level 3: 메타 학습 (Meta-Pattern Recognition)

```
누적 피드백:
1. "배열 + 더하기" = sum
2. "배열 + 평균" = average
3. "배열 + 최대" = max
4. "배열 + 최소" = min
  ↓
메타 패턴 인식:
"배열 + <연산> = <연산 함수>"
  ↓
결과: 새로운 연산이 들어와도 자동 패턴화
```

## 🤖 AI의 진화 과정

```
Phase 1 (초기): AI가 헤더를 일일이 검토
"이게 맞나? 아니가? 맞는 것 같긴 한데..."
(의존성 높음)

Phase 2 (학습 진행): AI가 패턴을 인식
"아, 배열 + 더하기는 항상 sum이네"
(패턴 인식)

Phase 3 (충분한 학습): AI가 자동 승인
"그냥 sum으로 해도 돼. 이미 알아"
(자율성 증가)

Phase 4 (진화): AI가 v2에 지시
"이건 속도 우선이야. 다르게 최적화해"
(상위 수준 제어)

→ AI가 점점 더 똑똑한 지휘관이 됨
```

## 💡 피드백 루프의 가치

### 사용자 관점 (v2 입장)
```
프로텍트 없음 (규칙 기반만)
→ 정확도 70%, 고정됨

프로텍트 있음 (피드백 학습)
→ 정확도 70% → 98% (진화함)

= 점점 더 똑똑한 어시스턴트
```

### AI 관점 (Claude 입장)
```
프로텍트 없음
→ 매번 명확한 입력 필요 (번거로움)

프로텍트 있음
→ 자유로운 입력 가능 (편함)
→ v2가 AI 의도를 더 잘 이해

= 점점 더 편한 개발 환경
```
